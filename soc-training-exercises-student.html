📋 Training Overview

These exercises provide hands-on experience with Security Operations Center (SOC) workflows using the four-screen simulation environment. They are organized into six sections: the three classic analyst tiers (Tier 1 Alert Triage → Tier 2 Incident Response → Tier 3 Threat Hunting), a Multidisciplinary section that applies the same SOC dashboards through the lenses of eight other fields — computer science, data science, business and risk management, criminal justice, psychology, healthcare, engineering, and social science — plus two modern-operations sections: Cloud & Identity attacks and AI-era threats.

Screen 1: SIEM Dashboard

Alert queue, severity filtering, MITRE ATT&CK heatmap, alert trends, SOC operational metrics (MTTD, MTTR, false-positive rate)

Screen 2: Network Monitor

Global threat map, traffic graphs, firewall logs, top talkers, port activity

Screen 3: Incident Response

Case management, IOC details, affected assets, response actions, playbooks, sector tagging (Healthcare/PHI, Finance/PCI, OT/Industrial)

Screen 4: Threat Intel & Endpoints

Threat feeds, endpoint status (incl. medical devices and OT/SCADA assets), vulnerability management, compliance scores

1 Alert Severity Assessment & Prioritization Beginner ⏱️ 20 min ▼

Learning Objectives

  • Understand alert severity levels and their meanings
  • Practice prioritizing alerts based on severity and context
  • Learn to filter alerts effectively in a SIEM dashboard

Screens Used

Screen 1: SIEM Dashboard

Scenario

You have just started your shift as a Tier 1 SOC analyst. The SIEM dashboard shows 47 unacknowledged alerts that accumulated overnight. You need to triage these alerts and determine the order in which they should be investigated.

Tasks

  • Open Screen 1 (SIEM Dashboard) and observe the alert queue for 2 minutes
  • Use the severity filters to view only Critical alerts. Document how many critical alerts exist.
  • For the following alert types, identify the appropriate severity level and explain why:

    Alert A: Failed SSH login from internal IP 10.0.1.45

    Alert B: Ransomware signature detected on endpoint WKS-042

    Alert C: Unusual DNS query to newly registered domain

    Alert D: Data exfiltration detected - 500MB upload to external IP

  • Create a prioritized list ranking these 4 alerts from highest to lowest priority
  • Explain the factors that influenced your prioritization decisions

Grading Rubric

Correct severity classifications (4 alerts)40 pts
Correct priority ordering30 pts
Clear justification of decisions20 pts
Professional documentation10 pts
2 False Positive Identification Beginner ⏱️ 25 min ▼

Learning Objectives

  • Distinguish between true positive and false positive alerts
  • Understand common causes of false positives in SOC environments
  • Learn when to tune detection rules vs. escalate alerts

Screens Used

Screen 1: SIEM Dashboard Screen 4: Threat Intel

Scenario

Your SOC has been receiving numerous alerts that appear to be false positives, creating alert fatigue. Management has asked you to analyze a sample of alerts and recommend which ones might need detection rule tuning.

Tasks

  • Review the following alerts and classify each as likely True Positive (TP) or False Positive (FP):

    Alert 1: Port scan detected from 10.0.5.100 (IT Security Scanner)

    Alert 2: Mimikatz signature detected in memory on DC-01

    Alert 3: Large file download from marketing@company.com (Google Drive sync)

    Alert 4: PowerShell encoded command execution on WKS-089

    Alert 5: Failed authentication from admin service account (nightly backup job)

  • For each False Positive identified, suggest a tuning recommendation
  • For each True Positive, describe the next investigation step
  • Check Screen 4 Threat Intel feed - does any intelligence correlate with the true positives?

💡 Tips for Identifying False Positives

  • Check if the source is a known authorized security tool
  • Look for patterns that match legitimate business processes
  • Verify if activity occurs during expected maintenance windows
  • Cross-reference with asset inventory and user roles

16 Detection Rule Authoring & Tuning Intermediate Computer Science ⏱️ 35 min ▼

Learning Objectives

  • Translate an informal security requirement into formal detection logic
  • Identify false-positive amplifiers in rule design
  • Apply precision/recall tradeoffs to rule tuning decisions

Screens Used

Screen 1: SIEM Dashboard Screen 4: Threat Intel

Scenario

Leadership wants a detection for "suspicious PowerShell execution." The first attempt was the rule EventID=4104 AND ScriptBlockText CONTAINS "Invoke", which generated 10,400 alerts/day with a 97% false positive rate. It was rolled back after 18 hours. You've been asked to propose a better rule.

Available signals: PowerShell ScriptBlock logging (4104), Process creation (4688), Parent process

Known legitimate uses: Admin scripts using Invoke-WebRequest, software deployment tools, SCCM client

Known malicious indicators: Base64-encoded commands, hidden windows, downloaded payload execution, uncommon parent processes

Tasks

  • Write a revised detection rule in pseudocode that lowers false positive rate to under 20%
  • Identify at least three conditions that are common in malicious PowerShell but rare in admin PowerShell
  • Describe a staged rollout plan (shadow mode, percentage rollout, monitored window)
  • Explain the precision/recall tradeoff of your rule: what do you gain, what do you give up?
  • Propose a metric and threshold for automatically rolling back your rule if it misbehaves

Grading Rubric

Rule logic correctness30 pts
Indicator selection quality25 pts
Rollout plan realism20 pts
Tradeoff articulation15 pts
Auto-rollback design10 pts

  • Where did the attacker get the voice and the project name? What does that imply about what your org should consider 'public attack surface'?

Learning Objectives

  • Verify AI-assistant triage verdicts against primary evidence
  • Recognize characteristic AI failure modes: hallucinated context, plausible-but-wrong reasoning
  • Define when AI triage should be trusted, reviewed, or overridden

Scenario

Your SOC has piloted an AI triage copilot. Overnight it dispositioned five alerts. Review its verdicts: (A) "Brute force vs VPN — ESCALATE: 300+ failures then a success from same IP" — logs confirm. (B) "Mass file rename on FS-HR-03 — CLOSE as FP: this server runs nightly batch-rename jobs" — the asset inventory lists NO scheduled jobs on FS-HR-03. (C) "Impossible travel for jchen — CLOSE: jchen is on the approved-VPN-exception list" — the list exists, jchen is on it, and the second IP is a corporate VPN egress. (D) "PowerShell with base64 on WKS-DEV-05 — CLOSE as FP: developers run encoded commands routinely" — true in general for this host, but the decoded command in the log contains Invoke-WebRequest to 91.219.237[.]14 followed by Start-Process. (E) "New admin account 'svc_backup2' — ESCALATE: no change ticket found" — correct, no ticket exists.

Verdict A: ESCALATE brute force — evidence cited matches logs

Verdict B: CLOSE rename storm, cites 'nightly batch job' — not in asset inventory

Verdict C: CLOSE impossible travel via exception list — list checks out

Verdict D: CLOSE encoded PowerShell as 'developer normal' — decoded payload fetches+executes from external IP

Verdict E: ESCALATE unauthorized admin account — correct

Tasks

  • Grade each verdict: correct, incorrect, or correct-conclusion-wrong-reasoning — and state the primary evidence that decides it
  • Verdicts B and D fail differently. Name each failure mode precisely (fabricated context vs. base-rate reasoning overriding instance evidence)
  • B closed a likely ransomware precursor. Write the incident note you'd file after re-opening it, including what the copilot's note said
  • Design the review policy: which alert classes may the copilot auto-close, which require human review, and what verdict-evidence standard must every AI disposition meet?
  • Data science lens: the team wants to measure the copilot before expanding it. Define the evaluation — metrics, ground truth source, and the asymmetric cost problem (a wrong CLOSE ≡ a wrong ESCALATE)
